# GoWith PRD: B站探店博主店铺地图集

版本：v0.1  
日期：2026-06-16  
状态：产品方案初稿  
产品定位：以 B站探店博主为索引的全国店铺地图与评价决策平台

## 1. 背景与机会

B站探店内容覆盖大量餐饮、本地生活、旅游目的地与城市消费场景，但当前信息主要分散在视频标题、简介、字幕、口播、弹幕、评论区和用户收藏夹里。用户在真实消费前，通常需要在 B站、大众点评、小红书、地图软件之间反复切换，才能回答几个关键问题：

- 这位博主去过哪些店？
- 这家店具体在哪里？
- 视频里到底推荐什么、避雷什么？
- 评论区反馈是否与博主观点一致？
- 这家店最近是否还值得去？
- 如果我信任某几个博主，能不能只看他们推荐过的店？

GoWith 要把“视频内容”转化为“地图上的可检索店铺情报”，用博主作为核心组织方式，建立从内容来源到店铺结论的证据链。

## 2. 产品目标

### 2.1 一句话目标

让用户按 B站探店博主发现全国店铺，并基于 AI 总结、视频证据、评论反馈和地图位置做消费决策。

### 2.2 阶段目标

MVP 阶段：

- 支持全国范围的店铺展示，但数据增长以“博主种子库”驱动，不按单城市封闭。
- 自动获取指定探店博主的视频列表、视频元信息和可用文本信息。
- 使用 AI 从视频标题、简介、标签、字幕、评论中抽取店铺、菜品、位置、评价观点。
- 建立店铺详情页、博主主页、首页推荐流和地图页。
- 每个 AI 结论可追溯到来源视频与文本证据。

上线阶段：

- 支持用户按博主、城市、品类、价格、人均、推荐强度、评价维度筛选。
- 支持自动化数据更新、店铺去重、POI 匹配和异常审核。
- 支持推荐系统、收藏、想去、去过、路线规划、用户反馈纠错。

长期阶段：

- 建立全国探店内容知识库。
- 建立博主口味画像、店铺可信度评分和用户个性化推荐。
- 支持商家纠错、创作者合作、专题榜单和商业化投放。

## 3. 用户与场景

### 3.1 目标用户

核心用户：

- 经常看 B站探店、美食、旅行、本地生活内容的用户。
- 对特定 UP主有信任偏好的用户。
- 到新城市旅行、出差、约会、聚餐前需要快速做决策的用户。

扩展用户：

- 探店博主：希望自己的探店内容被结构化展示，形成个人地图资产。
- 本地生活运营者：希望做城市榜单、路线专题、内容聚合。
- 商家：希望了解 B站内容中的店铺反馈，或提交纠错信息。

### 3.2 高频场景

1. 用户关注某个 B站探店博主，想看这个博主在全国去过哪些店。
2. 用户打开首页，希望看到附近或目标城市内值得去的店铺卡片。
3. 用户打开地图页，按博主筛选，看店铺在城市里的空间分布。
4. 用户进入店铺详情，想快速知道推荐菜、避雷点、人均、排队、交通、评论区反馈。
5. 用户旅行前选择多个博主，生成“想去清单”或一日路线。
6. 用户发现店铺位置、名称或状态错误，提交纠错。

## 4. 产品形态与多端策略

### 4.1 端形态

第一阶段建议顺序：

- Web H5 / PC 响应式：最快验证首页推荐、地图页、店铺详情和 SEO。
- 微信小程序：适合移动消费决策、位置授权、到店导航和分享。
- App 或 React Native：在收藏、路线、个性化推荐成熟后再投入。
- 管理后台：内部审核、数据任务、博主管理、店铺纠错、AI 结果校准。

### 4.2 设计风格

首页：

- 偏大众点评式店铺卡片流。
- 信息密度高，突出店名、图像、人均、位置、推荐理由、博主来源、AI 摘要。
- 内置推荐系统，不是纯搜索工具。

地图页：

- 地图为主体，支持全国缩放、城市聚焦、点位聚合、博主筛选。
- Pin 上展示店铺热度、博主数量、推荐强度或品类。
- 底部或侧边抽屉展示当前地图范围内的店铺列表。

博主页：

- 以“这个博主的探店地图”为核心资产。
- 展示博主去过的城市、品类偏好、推荐强度分布、最新视频、店铺集合。

## 5. 核心信息架构

一级导航建议：

- 首页：推荐店铺卡片流。
- 地图：全国店铺地图。

- 榜单：城市榜、品类榜、博主榜、近期热门。
- 我的：收藏、想去、去过、关注博主、路线。

核心对象：

- 博主 Creator
- 视频 Video
- 店铺 Shop
- POI 地点 PointOfInterest
- 评论 Comment
- 店铺评价 ShopReviewInsight
- 菜品 Dish
- 证据 Evidence
- 用户 User
- 收藏/想去 Collection
- 审核任务 ReviewTask

## 6. 核心功能需求

### 6.1 首页推荐

目标：用户一进来就能看到值得决策的店铺，而不是先被迫搜索。

功能：

- 店铺卡片流：
  - 店名、主图、品类、人均、城市/商圈、距离。
  - 推荐理由：由 AI 总结，限制为 1-2 句。
  - 博主来源：展示推荐过该店的 UP主头像/昵称。
  - 可信度标签：如“多博主提及”“近期更新”“评论区争议”“需复核”。
  - 推荐菜/避雷点。
  - 收藏、想去、导航、查看视频。
- 筛选：
  - 博主。
  - 城市/附近。
  - 品类。
  - 人均价格。
  - 推荐强度。
  - 最近更新时间。
  - 是否多博主共同提及。
- 推荐策略：
  - 初期：规则 + 内容画像。
  - 成熟后：召回、排序、重排三段式推荐。

验收标准：

- 用户可以在首页按博主筛选店铺。
- 每张卡片至少显示一个来源视频。
- 推荐理由不能脱离证据来源。

### 6.2 地图页

目标：让用户按空间位置浏览探店内容。

功能：

- 全国地图缩放。
- 店铺 pin 聚合。
- 按博主筛选。
- 按品类、价格、推荐强度筛选。
- 地图范围内店铺列表。
- 点击 pin 展示店铺浮层。
- 支持当前位置与目标城市切换。
- 支持跳转第三方地图导航。

关键交互：

- 地图缩放到全国/省级时展示聚合点。
- 缩放到城市/商圈时展示具体店铺 pin。
- 多个店铺位置过近时自动错位或列表聚合。
- 选择博主后，地图只显示该博主关联店铺。

验收标准：

- 全国范围数据不会导致地图首屏卡顿。
- 地图 pin 与列表筛选状态一致。
- 店铺详情可以从 pin 和列表进入。

### 6.3 博主页

目标：把每个探店博主沉淀为一个可浏览、可筛选、可收藏的探店地图。

功能：

- 博主基础信息：头像、昵称、B站 UID、粉丝量、简介、主页链接。
- 探店统计：店铺数、城市数、品类分布、最近更新时间。
- 博主地图：该博主关联店铺 pin。
- 博主店铺流：按时间、城市、推荐强度排序。
- 视频列表：展示被识别为探店内容的视频。
- 博主口味画像：
  - 常探品类。
  - 人均区间。
  - 城市偏好。
  - 常用评价词。
  - 推荐/吐槽比例。

验收标准：

- 用户可以关注博主。
- 用户可以只看某个博主推荐过的店。
- 每个店铺能展示该博主对应的视频证据。

### 6.4 店铺详情页

目标：让用户快速判断这家店值不值得去。

模块：

- 店铺基础信息：店名、地址、城市、商圈、品类、人均、营业状态。
- 来源视频：
  - 视频标题、UP主、发布时间、封面。
  - 跳转 B站原视频。
  - 相关片段时间点，若可获取。
- AI 总结：
  - 一句话结论。
  - 推荐菜。
  - 适合场景。
  - 优点。
  - 争议点。
  - 避雷点。
  - 排队/服务/环境/价格信息。
- 评论区反馈：
  - 正向高频观点。
  - 负向高频观点。
  - 争议观点。
  - 最近评论变化。
- 证据链：
  - 每个结论展示来源类型：标题、简介、字幕、评论、人工审核。
  - 展示置信度。
- 操作：
  - 收藏。
  - 想去。
  - 去过。
  - 导航。
  - 纠错。

验收标准：

- AI 总结必须有证据来源。
- 店铺位置必须有 POI 匹配记录。
- 低置信度店铺需展示“待确认”状态。

### 6.5 搜索与筛选

搜索对象：

- 店铺名。
- 博主名。
- 城市/商圈。
- 菜品。
- 视频标题。
- 评价关键词，如“排队少”“适合约会”“性价比高”。

筛选维度：

- 博主。
- 城市。
- 品类。
- 人均价格。
- 推荐强度。
- 评论情绪。
- 最近更新。
- 多博主共同推荐。
- 数据置信度。

### 6.6 用户系统

MVP 可选匿名，推荐上线前加入账号体系。

功能：

- 手机号/微信登录。
- 收藏店铺。
- 关注博主。
- 想去清单。
- 去过反馈。
- 纠错反馈。
- 个性化推荐偏好：城市、口味、价格、关注博主。

### 6.7 管理后台

目标：保证自动化系统可控。

模块：

- 博主种子库管理。
- 视频抓取任务管理。
- AI 抽取结果审核。
- 店铺 POI 匹配审核。
- 店铺合并/拆分。
- 低置信度结果队列。
- 用户纠错处理。
- 数据质量看板。
- 模型效果评估。

关键能力：

- 可查看每个店铺结论的证据。
- 可手动覆盖 AI 结果。
- 可回滚错误合并。
- 可标记视频非探店内容。

## 7. 数据获取与 AI 自动化

### 7.1 数据源优先级

优先级 1：官方开放平台、授权接口、创作者授权。

- 适合正式上线。
- 稳定性与合规性最好。
- 可能存在数据范围限制，需要申请权限或合作。

优先级 2：公开页面与公开接口的克制式增量采集。

- 适合内部验证和小规模灰度。
- 需要严格限流、缓存、robots/协议评估、数据最小化。
- 上线前需法务确认。

优先级 3：用户提交、博主提交、运营录入。

- 可用于补全数据和提升准确率。
- 适合建立高价值种子库。

不建议作为上线默认方案：

- 大规模无授权全量抓取。
- 绕过反爬、模拟登录批量拉取、采集非公开或需登录权限的数据。
- 未经授权缓存、重分发视频正文、评论全文或用户个人信息。

### 7.2 B站数据采集对象

博主层：

- UID。
- 昵称。
- 头像。
- 简介。
- 粉丝量。
- 主页链接。

视频层：

- BV号/AV号。
- 标题。
- 简介。
- 标签。
- 分区。
- 发布时间。
- 播放、点赞、收藏、投币、评论数。
- 封面图。
- 字幕或可用文本。
- 评论样本。

评论层：

- 评论内容。
- 点赞数。
- 发布时间。
- 楼中楼摘要。
- 情绪与主题分类。

注意：正式产品中不应默认展示评论用户的可识别个人信息，评论原文也应谨慎展示，优先展示 AI 聚合摘要。

### 7.3 自动化处理流程

1. 博主种子入库。
2. 定时获取博主视频列表。
3. 使用规则和模型判断视频是否为探店内容。
4. 抽取候选店铺名、城市、商圈、地址、菜品、人均、评价观点。
5. 调用地图 POI 服务做地点检索和地理编码。
6. 进行店铺实体去重与合并。
7. 对视频文本和评论样本做 AI 总结。
8. 生成店铺-博主-视频关系。
9. 生成证据链和置信度。
10. 低置信度结果进入人工审核。
11. 上线展示并进入增量更新。

### 7.4 AI 能力拆分

探店视频识别：

- 输入：标题、简介、标签、分区、评论摘要。
- 输出：是否探店、品类、城市候选、置信度。

实体抽取：

- 输入：标题、简介、字幕、评论。
- 输出：店名、分店名、菜品、地址线索、人均、场景、排队、服务、环境。

POI 匹配：

- 输入：店名、城市、商圈、地址线索。
- 输出：地图 POI ID、坐标、地址、匹配置信度。

评价总结：

- 输入：视频文本、评论样本、历史店铺观点。
- 输出：推荐理由、优点、缺点、争议点、适合人群、推荐菜、避雷点。

证据生成：

- 输入：模型输出和原始文本片段。
- 输出：每个结论的来源、证据片段、置信度。

质量检测：

- 检测幻觉、无证据结论、位置冲突、同名店误匹配、过期信息。

### 7.5 置信度模型

店铺结果置信度建议由以下因素组成：

- 店名抽取置信度。
- 城市/地址匹配置信度。
- POI 服务匹配分。
- 视频标题/简介/字幕中是否出现明确店名。
- 是否多个视频或多个博主共同提及。
- 评论区是否出现同名店反馈。
- 信息新鲜度。
- 人工审核状态。

展示策略：

- 高置信度：正常展示。
- 中置信度：展示但标注“位置待确认”或“信息待确认”。
- 低置信度：不进入公开推荐流，仅后台审核。

## 8. 推荐系统

### 8.1 推荐目标

首页不是简单时间流，而是帮助用户发现“更可能值得去”的店。

推荐排序目标：

- 用户可能感兴趣。
- 数据可信。
- 近期仍有效。
- 与用户位置或目标城市相关。
- 与用户关注博主相关。

### 8.2 MVP 推荐策略

规则排序：

- 关注博主优先。
- 附近/目标城市优先。
- 多博主共同推荐优先。
- 最近 180 天出现的视频优先。
- 高置信度优先。
- 评论负向争议过高降权。

内容画像：

- 用户偏好城市。
- 用户偏好品类。
- 用户价格区间。
- 用户关注博主。
- 用户收藏历史。

### 8.3 成熟推荐架构

召回层：

- 关注博主召回。
- 同城热门召回。
- 相似店铺召回。
- 相似用户收藏召回。
- 新热视频召回。

排序层：

- 点击率预测。
- 收藏率预测。
- 导航点击预测。
- 到店反馈预测。
- 负反馈风险。

重排层：

- 城市和品类多样性。
- 博主多样性。
- 避免连续相似店铺。
- 商业化内容标注与限频。

## 9. 技术栈分析

### 9.1 推荐总体架构

建议采用“Web 产品 + 数据管道 + AI 服务 + 地理空间服务 + 管理后台”的分层架构。

核心链路：

前端应用 -> API 网关 -> 业务服务 -> 数据库/搜索/缓存 -> 推荐服务  
数据采集任务 -> 数据清洗 -> AI 抽取/总结 -> POI 匹配 -> 审核 -> 发布索引

### 9.2 前端技术栈

推荐：

- Web：Next.js + React + TypeScript。
- 移动 H5：同一套 Next.js 响应式页面。
- 小程序：Taro 或原生微信小程序。
- App：React Native 或 Expo，后期再做。
- UI：Tailwind CSS + Radix UI / shadcn 风格组件，或自建设计系统。
- 地图：高德地图 JS API 或腾讯位置服务 Web 地图。

选择理由：

- Next.js 适合 SEO、店铺详情页、博主页、榜单页。
- TypeScript 有利于复杂数据模型维护。
- 首页卡片流和地图页可复用组件。
- 小程序适合国内本地生活场景，但地图能力需按平台适配。

风险：

- 地图 SDK 在 Web、小程序、App 的能力不完全一致。
- 全国点位渲染需要聚合、分片和视窗内加载，不能一次性加载全量点。

### 9.3 后端技术栈

推荐组合：

- API 服务：Node.js NestJS 或 Fastify。
- 数据与 AI Worker：Python FastAPI + Celery / Dramatiq。
- 工作流编排：Temporal、Airflow 或 Prefect。
- 数据库：PostgreSQL + PostGIS。
- 搜索：OpenSearch / Elasticsearch。
- 缓存：Redis。
- 消息队列：Kafka / RabbitMQ / Redis Streams。
- 对象存储：S3 兼容存储或云厂商 OSS/COS。

选择理由：

- PostGIS 适合全国 POI、距离计算、范围查询和地理索引。
- OpenSearch 适合店铺、博主、视频、菜品、评价关键词搜索。
- Python 生态更适合数据抓取、NLP、AI 编排和离线任务。
- Node.js/NestJS 适合业务 API、权限、后台管理和前端团队协作。

小团队简化版：

- Next.js 全栈 API + PostgreSQL/PostGIS + Redis + Python Worker。
- 暂缓 Kafka，用 Redis Streams 或数据库任务表支撑 MVP。
- 暂缓独立推荐服务，先用规则排序。

### 9.4 地图与 POI 服务选型

推荐优先考虑国内地图服务：

- 高德地图：Web JS API、地理编码、地点搜索、行政区划、导航跳转生态较完整。
- 腾讯位置服务：小程序生态、地点搜索、路线规划适配较好。
- 百度地图：POI 覆盖广，但不同端体验与坐标处理需评估。

不建议 MVP 以 Google Maps/Mapbox 作为中国主地图：

- 国内访问、合规、坐标体系、POI 覆盖与导航跳转都有不确定性。

技术注意点：

- 国内地图通常使用 GCJ-02 坐标系。
- 若采集源、地图源、数据库源混用 WGS-84/GCJ-02/BD-09，必须建立坐标转换与来源标记。
- 店铺同名分店很多，不能只用店名匹配，必须结合城市、商圈、地址、人均、评论线索。

### 9.5 AI 技术栈

建议设计为模型无关的 AI Gateway：

- 支持多个 LLM 供应商。
- 支持结构化输出 JSON Schema。
- 支持提示词版本管理。
- 支持输入输出留痕。
- 支持成本统计。
- 支持人工标注回流。

模型能力：

- LLM：视频文本理解、评价总结、结构化抽取。
- Embedding：店铺、视频、评价观点的语义检索。
- Reranker：搜索结果和推荐候选重排。
- ASR/OCR：仅在拥有授权或允许处理视频内容时使用。

存储：

- 结构化结果进 PostgreSQL。
- 文本索引进 OpenSearch。
- 向量可先用 pgvector，规模扩大后切 Milvus / Qdrant。

风险：

- AI 可能编造店铺、菜品、价格或结论。
- 评论区噪声高，容易受到玩梗、反讽、地域梗影响。
- 店名同名、分店同名、搬迁、闭店会导致位置错误。

控制策略：

- 所有结论必须有 Evidence。
- 低置信度不进推荐流。
- 店铺详情展示“信息更新时间”。
- 用户纠错和人工审核进入模型反馈闭环。

## 10. 合规与平台风险

### 10.1 B站数据使用

上线产品需重点确认：

- 是否通过官方开放平台、授权合作或创作者授权获取数据。
- 是否允许缓存和展示视频元信息、评论摘要、封面图。
- 是否允许对评论内容做自动化处理。
- 是否可以展示原评论文本或仅展示聚合摘要。
- 是否可以使用视频字幕、口播转写或切片时间点。

原则：

- 优先展示跳转 B站原视频，不搬运视频内容。
- AI 总结以摘要和观点聚合为主，避免复制大段原文。
- 评论区只做聚合洞察，不默认公开评论用户信息。
- 保留来源链接、发布时间和证据类型。
- 提供内容下架、删除和纠错机制。

### 10.2 地图与 POI 合规

上线前需确认：

- 地图服务商授权范围。
- POI 数据展示权限。
- 商业化使用限制。
- QPS、日调用量、缓存规则。
- 坐标和地图测绘相关要求。

### 10.3 AI 内容风险

需要避免：

- 将 AI 推断当成事实。
- 生成未经证据支持的店铺评价。
- 对商家作出过度负面、不可追溯的判断。
- 展示可能侵犯用户隐私的评论个人信息。

产品策略：

- 展示“AI 总结，仅供参考”。
- 重要结论可查看来源。
- 支持商家、用户、博主纠错。
- 高风险负面结论进入审核。

- 博主：100-300 个种子 UP主。

- 店铺：目标沉淀 1万-5万条候选店铺。
- 公开展示：只展示高置信度和审核通过数据。

## 14. 关键风险与应对

数据合规风险：

- 风险：B站数据采集、评论处理和内容摘要可能触及平台协议或版权边界。
- 应对：官方/授权优先；公开数据最小化；只展示摘要和来源链接；上线前法务评估。

店铺位置错误：

- 风险：同名店、分店、搬迁、闭店导致用户被导航到错误位置。
- 应对：多信号 POI 匹配；置信度；人工审核；用户纠错；店铺更新时间。

AI 幻觉：

- 风险：模型编造推荐菜、价格或评价。
- 应对：结构化输出、证据链、低置信度拦截、人工抽检。

全国地图性能：

- 风险：点位过多导致前端卡顿。
- 应对：服务端聚合、视窗查询、瓦片化、聚合 pin、分页列表。

推荐冷启动：

- 风险：用户初期没有行为数据。
- 应对：按博主关注、城市、品类和热门规则推荐。

数据更新成本：

- 风险：视频、评论、地图 API、AI 总结都产生调用成本。
- 应对：增量同步、缓存、优先处理高价值博主和热视频。
